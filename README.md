# CRM-LID1.1 · Лиды

Модуль **Leads** платформы AI-First Business OS — вход воронки CRM: приём лида из канала → квалификация (скоринг) → распределение на менеджера → конвертация в сделку (Lead Qualifier & Router, ФАЗА 1). Здесь же UI-прототип воронки лидов (`index.html`).

## Границы репозитория

**Здесь живёт всё про лиды:**
- модель `Lead` (своя схема БД `leads.*`), статусы new → qualified → routed → converted;
- движок `leads.py`: детерминированный скоринг (без AI), выбор воронки (новые/постоянные/тендер/проект), распределение по правилам (география, продукт, нагрузка);
- API `/leads`: приём, список, квалификация, распределение, конвертация;
- AI-обоснование квалификации (`ai.py`) — Итерация 1, за feature-flag, поверх детерминированного балла;
- обработчики событий: «кампания → лиды», обратная связь «сделка создана → ссылка у лида»;
- `index.html` — UI-прототип канбана лидов (5 стадий, BYN).

**Здесь НЕ живут (отдельные репозитории):**
- **CRM-блоки** — сделки, доска, карточка, документы/1С, Price Engine, сообщения → репозиторий `CRM`;
- **Тендеры** — своя воронка → отдельный репозиторий.

## Интеграция через шину событий (§2.4/§2.5)

Модуль не импортирует другие модули; все связи — событиями:
- `marketing.campaign.launched` → приём лидов из кампании;
- `leads.lead.converted` → модуль sales (репозиторий `CRM`) создаёт сделку;
- `sales.deal.created` (с `lead_id`/`deal_id`) → лид получает ссылку `deal_id`.

Поэтому конвертация здесь **не создаёт сделку напрямую** — это главное отличие после выноса из CRM.

## Структура

Код — содержимое модуля `modules/leads/` монорепо (импортирует `core.*`); сам по себе не запускается, подключается к ядру через `ModuleContract` (`module.py`). Подключение как git submodule — на следующем этапе. `migrations/` зарезервирована под миграции модуля.

## Правило коммитов

Любой функционал лидов коммитим сюда. CRM-блоки и тендеры — только в их репозиториях; здесь допустимы лишь точки интеграции через события шины.
